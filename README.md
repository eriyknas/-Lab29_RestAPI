## Краткое описание работы
В ходе лабораторной работы был создан полноценный RESTful API для управления списком задач. Были изучены принципы архитектуры REST, отличие Minimal API от Controller-based API, работа с DTO, правильное использование HTTP-статусов, а также инструменты тестирования API. Дополнительно были реализованы маршруты для поиска, фильтрации, сортировки и получения статистики.
## Список реализованных маршрутов
| HTTP Метод | Маршрут | Описание |
|:---:|---|:---:|
| **GET** | `/api/tasks` | Получить список всех задач |
| **GET** | `/api/tasks/{id}` | Получить задачу по её идентификатору |
| **GET** | `/api/tasks/search` | Поиск задач по заголовку или описанию |
| **GET** | `/api/tasks/priority/{level}` | Получить задачи с определённым приоритетом |
| **GET** | `/api/tasks/sorted` | Получить отсортированный список задач |
| **GET** | `/api/tasks/stats` | Получить статистику по задачам |
| **POST** | `/api/tasks` | Создать новую задачу |
| **PUT** | `/api/tasks/{id}` | Полностью обновить существующую задачу |
| **PATCH** | `/api/tasks/{id}/complete` | Переключить статус выполнения задачи |
| **DELETE** | `/api/tasks/{id}` | Удалить задачу |
## Итоговая таблица: ASP.NET Core Controller-based API
| Аспект | Описание в ASP.NET Core Controllers |
|:---|:---|
| **Маршруты** | Задаются атрибутами над методом |
| **Группировка маршрутов** | Класс-контроллер |
| **Базовый URL** | `[Route("api/[controller]")]` — `[controller]` |
| **Параметр пути** | (`int id`) - параметр метода |
| **Параметр запроса** |  `[FromQuery] bool? completed` |
| **Тело запроса** | `[FromBody] CreateTaskDto dto` |
| **Ответ 200** | `return Ok(data);` |
| **Ответ 201** | `return CreatedAtAction(...)` |
| **Ответ 404** | `return NotFound(...)` |
| **Ответ 204** | `return NoContent();` |
| **Типизация данных** | Строгая (C#) |
| **Документация** | Swagger - устанавливается отдельно |
## Главные выводы
1. REST — не протокол, а архитектурный стиль. Те же принципы работают с
любым языком и фреймворком.
2. Контроллер в ASP.NET Core = Router в Express, только с автоматической
документацией и строгой типизацией.
3. DTO защищает API от некорректных данных: клиент передаёт только то, что
сервер разрешает принять.
4. Swagger UI позволяет тестировать API без Postman и без написания тестового
JavaScript-кода.
5. Правильные HTTP-статусы — часть «контракта» API. Клиент должен
понимать, что произошло, по коду ответа.
